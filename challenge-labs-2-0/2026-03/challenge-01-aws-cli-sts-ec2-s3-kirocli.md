# Challenge 01: AWS CLI, STS, EC2, S3 and KiroCLI

## Challenge objective
Manage an EC2 instance using AWS CLI, create a local directory and synchronise its files to S3, using an AI assistant and temporary credentials.

### Mandatory constraints
- The process must run through a specific IAM role with permissions for EC2 and S3.
- The IAM user configured in AWS CLI must not have any policy attached directly.
- Two scripts must be created with AI assistance:
  - Launch EC2.
  - Sync files to S3.
- A final diagnostic report must prove that resources were launched and accessed correctly.

## Scope
- AWS CLI profile configuration
- STS session token and AssumeRole flow
- EC2 lifecycle operations through tags
- S3 daily operations (`ls`, `cp`, `presign`, `sync`, `rm`)
- AI-assisted command generation with KiroCLI

## What was done in this challenge
This challenge was executed as a security progression:
- Started with long-term credentials configured in AWS CLI.
- Moved to temporary credentials generated by STS.
- Migrated to role-based access via `AssumeRole`, removing direct user permissions.
- Executed EC2 and S3 operations fully through CLI under the assumed role.
- Used KiroCLI to generate reusable scripts for repeatable operations.
- Validated the setup with command-line diagnostics to confirm access and provisioning.

## Prerequisites
- AWS CLI v2 installed
- Docker Engine installed (for Docker volume workflow)
- Base profile configured (example: `awscli`)
- A role available for assumption (example: `role-team-dev`)
- Region, VPC, subnet, key pair, and security group already created

### Optional IAM validation (recommended)
If your credentials allow IAM read actions, verify that the CLI user has no direct policies attached:


## 1) Configure and verify a base profile
Use a named profile to avoid mixing credentials with the default account.

```bash
# Configure credentials and default region/output for the profile
aws configure --profile awscli

# Verify resolved credentials and configuration sources
aws configure list --profile awscli

# Basic permission check with explicit profile
aws sts get-caller-identity --profile awscli
aws s3 ls --profile awscli
```

## 2) Generate temporary credentials with STS session token
Use short-lived credentials for safer local operations.

```bash
# Request and export temporary credentials valid for 15 minutes (no jq required)
read AWS_ACCESS_KEY_ID AWS_SECRET_ACCESS_KEY AWS_SESSION_TOKEN < <(
  aws sts get-session-token \
    --duration-seconds 900 \
    --profile awscli \
    --query 'Credentials.[AccessKeyId,SecretAccessKey,SessionToken]' \
    --output text
)
export AWS_ACCESS_KEY_ID AWS_SECRET_ACCESS_KEY AWS_SESSION_TOKEN

# Confirm caller identity with temporary credentials
aws sts get-caller-identity
```

## 4) Manage EC2 instances via CLI
Use tags to target instances safely.

```bash
# Reusable variables
export AWS_REGION=us-east-1
export KEY_NAME=formacaoawskey
export SECURITY_GROUP_ID=sg-04160637fc1b1467b
export SUBNET_ID=subnet-034e91ffcbe378258
```

### Launch one tagged instance
```bash
aws ec2 run-instances \
  --image-id ami-02dfbd4ff395f2a1b \
  --instance-type t3.micro \
  --key-name "$KEY_NAME" \
  --security-group-ids "$SECURITY_GROUP_ID" \
  --subnet-id "$SUBNET_ID" \
  --region "$AWS_REGION" \
  --count 1 \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=group,Value=automation-cli}]'
```

### List instances by tag
```bash
aws ec2 describe-instances \
  --filters "Name=tag:group,Values=automation-cli" \
  --query "Reservations[*].Instances[*].[InstanceId,State.Name,InstanceType]" \
  --output table
```

### Stop instances by tag
```bash
instance_ids=$(aws ec2 describe-instances \
  --filters "Name=tag:group,Values=automation-cli" \
  --query "Reservations[*].Instances[*].InstanceId" \
  --output text)

aws ec2 stop-instances --instance-ids $instance_ids --output table
```

### Start instances by environment tag
```bash
instance_ids=$(aws ec2 describe-instances \
  --filters "Name=tag:environment,Values=prod" \
  --query "Reservations[*].Instances[*].InstanceId" \
  --output text)

aws ec2 start-instances --instance-ids $instance_ids --output table
```

### Terminate instances (careful)
```bash
# Run this only when cleanup is intended
aws ec2 terminate-instances --instance-ids $instance_ids --output table
```

## 5) Operate S3 in a Docker volume
Reference: https://docs.aws.amazon.com/cli/latest/reference/s3/

### 5.1) Store documents in a Docker volume and sync to S3
Use this flow when files are produced inside containers and need to be published to S3 with a local audit copy.

```bash
# 1) Create a named Docker volume for challenge documents
docker volume create docs_volume

# 2) Create sample files inside the volume (simulates container output)
docker run --rm -v docs_volume:/docs alpine sh -c \
  'echo "Challenge evidence" > /docs/report.txt && date -u > /docs/generated-at.txt'

# 3) Verify files inside the volume
docker run --rm -v docs_volume:/docs alpine ls -lah /docs

# 4) Mirror volume content to a local folder for auditing/versioning
mkdir -p tmp/docker-docs
docker run --rm \
  -v docs_volume:/from:ro \
  -v tmp/docker-docs:/to \
  alpine sh -c 'cp -rv /from/. /to/'

# 5) Preview S3 synchronisation before applying changes
aws s3 sync tmp/docker-docs s3://course-automation-s3-cli/docker-docs --dryrun

# 6) Execute the synchronisation
aws s3 sync tmp/docker-docs s3://course-automation-s3-cli/docker-docs
```

Optional: synchronise directly from the Docker volume (no host copy).

```bash
# Dry run direct sync from the mounted volume
docker run --rm \
  -v docs_volume:/docs:ro \
  -v "$HOME/.aws:/root/.aws:ro" \
  amazon/aws-cli:latest \
  s3 sync /docs s3://course-automation-s3-cli/docker-docs --dryrun

# Execute direct sync from the mounted volume
docker run --rm \
  -v docs_volume:/docs:ro \
  -v "$HOME/.aws:/root/.aws:ro" \
  amazon/aws-cli:latest \
  s3 sync /docs s3://course-automation-s3-cli/docker-docs
```

### 5.2) Operate S3 with safe command patterns

```bash
# Keep a single local workspace for audit/version control
mkdir -p tmp/docker-docs

# List all buckets in the current account
aws s3 ls

# List objects in the target bucket
aws s3 ls s3://course-automation-s3-cli

# Generate pre-signed URLs (default expiry is 3600 seconds)
aws s3 presign s3://course-automation-s3-cli/arquivo1.txt
aws s3 presign s3://course-automation-s3-cli/arquivo1.txt --expires-in 10

# Copy operations between local audit folder and S3
aws s3 cp tmp/docker-docs/arquivo1.txt s3://course-automation-s3-cli/
aws s3 cp s3://course-automation-s3-cli/arquivo2.txt tmp/docker-docs/

# Preview operations before applying changes
aws s3 cp tmp/docker-docs/arquivo1.txt s3://course-automation-s3-cli/ --storage-class STANDARD_IA --dryrun
aws s3 sync tmp/docker-docs s3://course-automation-s3-cli --storage-class STANDARD_IA --dryrun
aws s3 rm s3://course-automation-s3-cli/ --recursive --dryrun
```

### 6.4 Use prompts to generate scripts
Prompt examples for KiroCLI:

```text

- The script must launch one EC2 instance with variables: AMI_ID, KEY_NAME, SECURITY_GROUP_ID, SUBNET_ID, AWS_REGION.
- Add inline comments explaining each command.
- Generate a Bash script named scripts/02-sync-to-s3.sh using set -euo pipefail.
- The script must accept <local-dir> and <s3-uri>, run aws s3 sync --dryrun first, then run the real sync.
- Add usage examples and error handling for missing arguments.
```

### 6.5 Save, review, and execute generated scripts
After KiroCLI returns script content, save files under `scripts/` and make them executable.

```bash

# Syntax check before execution
bash -n scripts/01-launch-ec2.sh
bash -n scripts/02-sync-to-s3.sh

# Execute script 1 (launch EC2)

KEY_NAME=formacaoawskey \
SECURITY_GROUP_ID=sg-xxxx \
SUBNET_ID=subnet-xxxx \

# Execute script 2 (sync local folder to S3)
./scripts/02-sync-to-s3.sh tmp/docker-docs s3://course-automation-s3-cli
```

## 7) Required script deliverables
The challenge requires two scripts generated with AI assistance.

### Script 1: launch EC2
Suggested file: `challenge-01-aws-cli-sts-ec2-s3-kirocli/scripts/01-launch-ec2.sh`

```bash
#!/usr/bin/env bash
set -euo pipefail

AWS_REGION="${AWS_REGION:-us-east-1}"
AMI_ID="${AMI_ID:?AMI_ID is required}"
INSTANCE_TYPE="${INSTANCE_TYPE:-t3.micro}"
KEY_NAME="${KEY_NAME:?KEY_NAME is required}"
SECURITY_GROUP_ID="${SECURITY_GROUP_ID:?SECURITY_GROUP_ID is required}"
SUBNET_ID="${SUBNET_ID:?SUBNET_ID is required}"

aws ec2 run-instances \
  --image-id "$AMI_ID" \
  --instance-type "$INSTANCE_TYPE" \
  --key-name "$KEY_NAME" \
  --security-group-ids "$SECURITY_GROUP_ID" \
  --subnet-id "$SUBNET_ID" \
  --region "$AWS_REGION" \
  --count 1 \
  --tag-specifications 'ResourceType=instance,Tags=[{Key=group,Value=automation-cli}]'
```

### Script 2: sync local folder to S3
Suggested file: `challenge-01-aws-cli-sts-ec2-s3-kirocli/scripts/02-sync-to-s3.sh`

```bash
#!/usr/bin/env bash
set -euo pipefail

LOCAL_DIR="${1:?Usage: $0 <local-dir> <s3-uri>}"
S3_URI="${2:?Usage: $0 <local-dir> <s3-uri>}"

# First show what would change
aws s3 sync "$LOCAL_DIR" "$S3_URI" --dryrun

# Then execute for real
aws s3 sync "$LOCAL_DIR" "$S3_URI"
```

## Validation checklist
- `aws sts get-caller-identity` returns expected principal before and after AssumeRole.
- S3 access fails for the base user after policy removal and succeeds after role assumption.
- EC2 instances are correctly filtered by tag and lifecycle commands execute as expected.
- S3 dry-run commands show expected file operations before real execution.

## Common issues
- `AccessDenied`: verify role trust policy and current caller identity.
- `InvalidClientTokenId`: credentials expired; request new STS credentials.
- Empty `instance_ids`: check region, tag key/value, and filter syntax.
- `Unknown output type: JSON`: set output as lowercase `json` in AWS CLI config or use `--output json`.
